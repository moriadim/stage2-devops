# 📕 On-Call Runbook: Stage 3 Observability & Alerts

## 📋 Summary
This runbook is designed for the **On-Call SRE / DevOps Engineer**. It provides standard operating procedures (SOPs) for responding to alerts generated by our Python log-watcher. Our goal is to minimize Mean Time to Recovery (MTTR) by clearly defining triage and mitigation steps for failovers and high error rates.

---

## 🚨 Alert: Failover Detected

**Context:** The log-watcher observed a change in the active routing pool (e.g., `blue` → `green`). 
**Blast Radius:** The failover mechanism worked, so users *might* not be experiencing downtime, but one of our application pools has completely failed or was manually taken offline.

### 🛠️ Immediate Mitigation Steps:
1. **Verify the Current State:**
   Check recent Nginx logs to confirm which pool is currently handling traffic and returning `200 OK`.
   ```bash
   make logs
   ```
   Or explicitly verify the active proxy: `docker logs nginx_proxy`.

2. **Inspect the Failing Container (Health Check Failure):**
   Determine *why* the previous pool failed. Check for OOMKilled events, exception stack traces, or bad configurations.
   ```bash
   docker ps -a
   docker logs app_blue   # Replace with the newly inactive pool
   ```

3. **Manual Health Check:**
   Fire a manual request to the inactive container to see if it responds to health probes.
   ```bash
   curl http://<container_ip>:<port>/healthz
   ```

4. **Resolution:** 
   If it was a crash, restart the container (`docker restart <container_name>`). If it was a planned deployment, ignore this alert.

> **💡 Planned Maintenance?** If you are performing a manual traffic switch, silence these alerts temporarily:
> Set `MAINTENANCE_MODE=true` in `.env` and restart the watcher (`docker restart alert_watcher`).

## 🚨 Alert: High Upstream 5xx Rate

**Context:** The system is currently returning an unacceptable level of server errors (5xx) back to our users. The error rate has breached our `ERROR_RATE_THRESHOLD` (default: 2%) over the defined `WINDOW_SIZE`.
**Blast Radius:** Users are experiencing active failures, timeouts, or application crashes.

### 🛠️ Immediate Mitigation Steps:
1. **Identify the Culprit Release:**
   Check the Slack alert details. Which pool is active? What is the `X-Release-Id` attached to the errors?

2. **Check the Logs:**
   Immediately investigate the application logs for the active pool to find the root cause (e.g., Database connection lost, missing environment variable).
   ```bash
   docker logs --tail 100 app_green  # Replace with the currently active pool
   ```

3. **Rollback / Manual Failover:**
   If the active pool (e.g., Green) is fundamentally broken due to a bad release, manually switch the traffic back to the healthy standby pool (e.g., Blue) by adjusting the Nginx upstream configuration or stopping the Green container to force Nginx to use Blue.
   ```bash
   docker stop app_green
   ```

## 📖 How to Read Slack Alerts
- **Time:** The exact UTC timestamp the anomaly hit the log file.
- **Window & Errors:** Helps you understand the sample size. "10 errors in 200 requests" means a 5% failure rate. If the window is too small, it might be a transient network blip.
- **Recent Release:** The `X-Release-Id` helps you tie the failure directly to a recent CI/CD deployment or code commit.
